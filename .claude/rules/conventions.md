# 프로젝트 작업 규칙 (사용자 지정)

## PR / 커밋
- PR 제목·본문은 **한국어**로 작성한다.
- PR 본문에 "Generated with Claude Code" 등 Claude 관련 문구를 넣지 않는다.
- 유저에게 알릴 만한 변경이면 PR 본문에 `## 공지` H2 섹션을 넣는다 — master 머지·배포
  성공 시 그 섹션만 디스코드 공지 채널에 자동 게시된다 (`deploy/README.md` 참고,
  다음 `## ` 헤딩 전까지가 공지 내용). 공지할 게 없으면 섹션을 생략한다.
- 관리자/개발자용 변경(업로드 UX, 내부 리팩토링 등)은 공지에서 제외하고
  유저 체감 변화(새 기능, 채점 정책, 버그 수정)만 공지에 담는다.

## 채점 정보 노출 정책 (백준 기준)
- 시간/메모리는 **정답(ACCEPTED)일 때만** 노출한다.
- 테스트케이스 **개수는 비공개** — 채점 중에는 진행률 %만 보여준다.
- 비공개 테스트 데이터를 탐침할 수 있는 정보 노출을 피한다.

## 문제/테스트케이스
- 대용량 테스트케이스는 원시 데이터 대신 **생성기 양식**(`.md`의 `<!-- @generator -->`,
  시드 기반)으로 관리한다. 시드가 같으면 데이터가 바이트 단위로 재현된다.
- 생성기에는 가능하면 `~~~validator`(독립 구현 두 번째 정답)를 붙여 케이스를 검증한다.
- nginx 본문 제한(1MB)은 올리지 않는다 — 업로드는 앱 레벨(생성기/분할 업로드)로 해결한다.
- 지문 이미지는 S3(`deploy/aws-s3-images.md`)로 올리되, 간단한 도형·다이어그램은
  인라인 SVG를 권장한다 (가볍고 확대해도 선명).

## 작업 방식
- 방향이 갈리는 결정은 질문을 최소화하고 합리적 기본값으로 진행한 뒤 사후 보고한다.
- 인프라(Judge0 박스 등) 수동 작업이 필요한 변경은 "박스에서 할 일"과
  "PR로 자동 반영되는 것"을 구분해 문서화한다 (`deploy/judge0/README.md` 예시).
